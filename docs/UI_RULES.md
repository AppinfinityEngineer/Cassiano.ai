# ThoughtSnap Labs UI Rules

## Product experience

- The primary purpose and next action must be obvious.
- Onboarding must demonstrate value quickly and avoid unnecessary questions.
- Every screen must support loading, empty, error, success, and disabled states where applicable.
- Never ship dead controls, fake data presented as real, or inaccessible interactions.

## Visual system

- Use design tokens for colour, typography, spacing, radius, shadows, and motion.
- Reuse shared components instead of duplicating page-level styling.
- Maintain clear hierarchy, generous spacing, readable line lengths, and consistent alignment.
- Responsive layouts must be intentionally designed for supported widths.
- Use motion sparingly to clarify state or add delight; respect reduced-motion preferences.

## Accessibility

- Use semantic structure and labelled controls.
- Ensure complete keyboard operation and visible focus states.
- Associate validation messages with fields and announce important state changes.
- Maintain sufficient contrast and usable target sizes.
- Do not communicate meaning through colour alone.

## Review requirements

Meaningful UI branches must include screenshots at representative mobile and desktop widths, plus notes covering accessibility, loading, empty, and error behaviour.
